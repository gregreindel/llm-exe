### 1. Mock LLM name is wrong in CLAUDE.md

- **What I tried**: `useLlm("openai.chat-mock")` as documented in CLAUDE.md ("Mock LLM providers: `openai.chat-mock`")
- **What happened**: `LlmExeError: Invalid provider: openai.chat-mock`
- **What I expected**: A working mock LLM
- **Actual key**: `openai.chat-mock.v1` — you need the `.v1` suffix
- **Severity**: confusing
- **File/line**: CLAUDE.md references `openai.chat-mock`, actual config at `src/llm/config/openai/index.ts:46`
- **Note**: Similarly, `openai.mock` (also referenced in CLAUDE.md) doesn't exist. Only `openai.chat-mock.v1` is in the config.

### 2. Invalid provider error messages don't suggest alternatives

- **What I tried**: `useLlm("openai")` and `useLlm("openai.gpt4o")` (typo, missing hyphen)
- **What happened**: `Invalid provider: openai.gpt4o` — no suggestions, no list of valid providers
- **What I expected**: "Invalid provider: openai.gpt4o. Did you mean openai.gpt-4o?" or at minimum a link to docs listing valid providers
- **Severity**: rough-edge
- **File/line**: `src/llm/config.ts:33` — the error is thrown with just the invalid name, no help

### 3. `createParser("xml")` silently falls back to StringParser — NO ERROR

- **What I tried**: `createParser("xml")` — a parser type that doesn't exist
- **What happened**: No error. Returns a `StringParser` that just passes input through unchanged.
- **What I expected**: A thrown error like "Invalid parser type: xml. Valid types are: string, json, boolean, number, listToArray, listToKeyValue, markdownCodeBlock, stringExtract"
- **Severity**: genuine-bug
- **Note**: This is dangerous. If I typo a parser name (e.g. `"jsonn"` instead of `"json"`), I get no error — my code just silently doesn't parse anything. I'd spend 30 minutes debugging why my JSON output isn't being parsed before realizing the parser silently fell back to string.

### 4. Boolean parser only accepts literal "true"

- **What I tried**: `createParser("boolean")` then `.parse("yes")`, `.parse("no")`, `.parse("true")`
- **What happened**: Only `"true"` returns `true`. Both `"yes"` and `"no"` return `false`.
- **What I expected**: A boolean parser that handles common LLM boolean outputs: "yes"/"no", "true"/"false", "1"/"0"
- **Severity**: rough-edge
- **File/line**: `src/parser/parsers/BooleanParser.ts:17-20` — only checks `clean === "true"`, everything else returns `false`
- **Note**: If I tell an LLM "respond with yes or no" and pipe it through the boolean parser, "yes" becomes `false`. That's a foot-gun.

### 5. listToArray doesn't strip numbered list prefixes consistently

- **What I tried**: `createParser("listToArray").parse("1. First\n2. Second")`
- **What happened**: Returns `["1. First", "2. Second"]` — keeps the numbering
- **What I expected**: `["First", "Second"]` — stripped of the list numbering
- **Severity**: rough-edge
- **Note**: Dashes ARE stripped (`"- First"` -> `"First"`), but numbered lists (`"1. First"`) and bullets (`"* First"`) are NOT stripped. Inconsistent behavior across the three most common list formats LLMs produce.

### 6. Dialogue API naming inconsistent with Prompt API

- **What I tried**: `createDialogue()` then `d.addUserMessage("Hello")` — guessing based on prompts which use `addUserMessage`
- **What happened**: `TypeError: d.addUserMessage is not a function`. The actual method is `setUserMessage`.
- **What I expected**: Same method names as prompts (`add*` prefix)
- **Severity**: confusing
- **Note**: Prompts use `add*` (addUserMessage, addAssistantMessage). Dialogue uses `set*` (setUserMessage, setAssistantMessage). Same domain concept, different naming. Also `set*` implies replacing, not appending — misleading.

### 7. Missing template variables silently render as empty string

- **What I tried**: `createChatPrompt<{ name: string; age: number }>("Hello {{name}}, you are {{age}}").format({ name: "Greg" } as any)`
- **What happened**: Renders as `"Hello Greg, you are "` — silent empty string for missing `age`
- **What I expected**: At minimum a console warning
- **Severity**: rough-edge
- **Note**: Standard Handlebars behavior, but for an LLM toolkit where prompt accuracy matters, silently dropping variables could lead to confusing LLM output that's hard to debug.
