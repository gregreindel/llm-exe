### 1. Mock LLM shorthand not documented correctly

- **Fix**: Must use `useLlm("openai.chat-mock.v1")` — the `.v1` suffix is required
- **Severity**: confusing
- **File/line**: CLAUDE.md line mentioning mock providers; `src/llm/config/openai/index.ts:46` (the actual key is `openai.chat-mock.v1`)

### 2. JSON parser silently returns `{}` for unparseable input

- **What I tried**: `createParser("json").parse("This is not JSON at all")`
- **What happened**: Returns `{}` (empty object) — no error, no warning
- **What I expected**: An error, or at minimum some signal that parsing failed. In a production pipeline, this means garbage LLM output silently becomes an empty object, which downstream code may treat as valid data.
- **Severity**: genuine-bug
- **File/line**: `src/utils/modules/json.ts:15` — `maybeParseJSON` returns `{} as Expected` for any falsy or unparseable input. The `helpJsonMarkup` call in `JsonParser.parse()` at `src/parser/parsers/JsonParser.ts:20` doesn't help because the input isn't markdown-wrapped.
- **Note**: Same behavior for empty string input. This is particularly dangerous because with a schema but without `validateSchema: true`, the schema enforcement just strips keys, so `{}` passes through silently even with a schema that has required fields.

### 3. JSON parser schema enforcement vs. validation is confusing

- **What I tried**: `createParser("json", { schema: {..., required: ["name"]} })` (without `validateSchema: true`)
- **What happened**: `parse('{"age": 25}')` returns `{}` — strips keys not in schema but does not check `required`
- **What I expected**: Either validate by default, or document that `validateSchema: true` is needed for required field checks
- **Severity**: confusing
- **Note**: The two-step process (enforce then optionally validate) is not obvious. Most users would expect that passing a schema with `required` fields would... require those fields. The `validateSchema` option defaults to `undefined`/falsy, meaning schemas are half-enforced by default.

### 4. Boolean parser only recognizes literal "true"

- **What I tried**: `createParser("boolean").parse("yes")`, `.parse("1")`, `.parse("no")`
- **What happened**: All return `false`. Only the exact string `"true"` (case-insensitive) returns `true`.
- **What I expected**: Common truthy values like "yes", "1", "y" to return `true`, and "no", "0", "n" to return `false`
- **Severity**: rough-edge
- **File/line**: `src/parser/parsers/BooleanParser.ts:17` — only checks `clean === "true"`
- **Note**: LLMs commonly respond with "Yes" or "No" to boolean questions. A boolean parser that only accepts the literal word "true" is surprising. An LLM saying "Yes, that's correct" would parse as `false`.

### 5. Number parser returns -1 for non-numeric input

- **What I tried**: `createParser("number").parse("not a number")`
- **What happened**: Returns `-1`
- **What I expected**: An error, or `NaN`, or `null`. Returning `-1` is a valid number that could be confused with actual data. There's no way for the caller to distinguish "the LLM said -1" from "parsing failed."
- **Severity**: rough-edge
- **File/line**: `src/parser/parsers/NumberParser.ts:14` — `match ? toNumber(match[0]) : -1`
- **Suggestion**: Throw an error or return `NaN` when no number is found.

### 6. stringExtract parser returns empty string on no match (no error)

- **What I expected**: An error or at least a documented sentinel value. In production, empty string could be a valid value in some contexts.

## What Worked Well

- **Mock LLM** (once I found the right key) works great for testing without API keys
- **Hook system** is solid: `on`, `off`, `once`, `clearHooks`, `getHookCount` — good lifecycle management for long-running processes
- **traceId** propagation through hooks and metadata is clean
- **Error metadata in onError hook** is thorough — includes input, handlerInput, handlerOutput, error, errorMessage, timing
- **Execution counter** is useful for monitoring
- **Retry with exponential backoff** via `backOff` in `requestWrapper.ts` is production-ready, with configurable `numOfAttempts`, `maxDelay`, `jitter`, and `doNotRetryErrorMessages`
- **Hook limit** (maxHooksPerEvent = 100) prevents unbounded memory growth — good defensive design

## Production Readiness Assessment

**Concerns:**
- Silent parser failures (findings 2, 5, 6) are the biggest risk. In production, you'd want to know when the LLM returned garbage, not get an empty object silently.
- The `validateSchema: true` being opt-in (finding 3) means most users will ship with half-validated schemas.
- The timeout error message (finding 10) would make debugging production issues harder.

**Positives:**
- Retry/backoff is well-implemented
- Hook system has proper memory management
- Metadata/tracing support is production-grade
- No obvious memory leaks in the core path (executor creates metadata per call, doesn't accumulate)
