## Findings

### 1. Executor result type is `any` — type inference doesn't reach the consumer

- **What I tried**: Created a full pipeline with `defineSchema`, `createParser("json", { schema })`, `createChatPrompt`, `useLlm`, and `createLlmExecutor`. Then checked if `executor.execute()` returns a typed result.
- **What happened**: The result is typed as `any`. I can assign it to `number`, access `.bogus`, or do anything — TypeScript doesn't complain. I verified this with `@ts-expect-error` directives that were flagged as "unused" (meaning the expected error didn't happen).
- **What I expected**: CLAUDE.md says "The type safety on LLM responses is what makes llm-exe valuable" and "Full type inference must flow from prompt input -> parser output -> executor return type." The parser's `parse()` method also returns `any` when called directly, even with a schema.
- **Severity**: genuine-bug
- **File/line**: The dist types look correct in theory (`BaseParserWithJson` uses `FromSchema<S>`), but when you actually write consumer code, the type resolves to `any`. The issue might be in how `Narrow<T>` from `defineSchema` interacts with the `createParser` generics, or how the overloads infer `S`.
- **Notes**: This is the single biggest issue I found. The README and CLAUDE.md both emphasize type safety as the core value prop. As a beginner, I would have trusted that types "just work" and been very confused when my IDE showed `any` everywhere. I tested with `strict: true` in tsconfig.

### 2. `listToJson` parser breaks on colons in values

- **What I tried**: `createParser("listToJson").parse("URL: https://example.com\nTime: 10:30")`
- **What happened**: Returns `{ url: "https", time: "10" }` — everything after the first colon is lost
- **What I expected**: `{ url: "https://example.com", time: "10:30" }`
- **Severity**: genuine-bug
- **File/line**: `src/parser/parsers/ListToJsonParser.ts:18` — uses `line.split(":")` which splits on ALL colons, then only takes `[0]` and `[1]`
- **Notes**: Should use `line.split(":")` with a limit or split only on the first colon. URLs, timestamps, and many common values contain colons. This will silently corrupt data.

### 3. `listToJson` silently camelCases keys

- **What I tried**: `createParser("listToJson").parse("First Name: Alice\nLast Name: Smith")`
- **What happened**: Returns `{ firstName: "Alice", lastName: "Smith" }`
- **What I expected**: Either the original keys (`"First Name"`, `"Last Name"`) or at least documentation that keys get camelCased
- **Severity**: confusing
- **File/line**: `src/parser/parsers/ListToJsonParser.ts:19` — uses `camelCase(key)`
- **Notes**: Not necessarily wrong, but nowhere documented. If I'm building a schema expecting `"first_name"` or `"First Name"`, I'd be very confused why validation fails.

### 4. `listToJson` overwrites duplicate keys (not an array)

- **What I tried**: `createParser("listToJson").parse("Name: Alice\nAge: 25\n\nName: Bob\nAge: 30")`
- **What happened**: Returns `{ name: "Bob", age: "30" }` — Alice is gone
- **What I expected**: Either an array of objects, or an error, or at least documentation explaining it only handles single-item lists
- **Severity**: rough-edge
- **Notes**: Previous session noted this too. The parser name "listToJson" implies it handles lists, but it actually just creates a single flat object from key:value lines.

### 5. `markdownCodeBlocks` (plural) vs `markdownCodeBlock` (singular) — works but undocumented

- **What I tried**: Used both parsers on input with two fenced code blocks
- **What happened**: `markdownCodeBlocks` returns an array of `{ language, code }` objects. `markdownCodeBlock` returns just the first one (a single object, not array).
- **What I expected**: This actually makes total sense! But nowhere in the docs is this distinction explained.
- **Severity**: confusing
- **Notes**: The README only mentions `markdownCodeBlock` (singular). CLAUDE.md lists both but doesn't explain the difference. A one-liner in the docs would save beginners a lot of confusion.

### 6. `replaceStringTemplate` parser works but purpose is unclear

- **What I tried**: `createParser("replaceStringTemplate").parse("Hello {{name}}!", { name: "World" })`
- **What happened**: Returns `"Hello World!"` — it does Handlebars substitution on the LLM output
- **What I expected**: Wasn't sure what to expect — this parser isn't documented anywhere
- **Severity**: confusing
- **Notes**: This is actually cool — you can template the LLM's output before returning it. But I can't imagine when I'd use it without an example. The second argument (attributes) comes from the executor context, which means in a pipeline the LLM output gets Handlebars-processed with the original input data. Zero documentation on when/why you'd want this.

### 7. `createState()` takes no arguments but docs are unclear

- **What I tried**: `createState("myState")` (passing a name like `createDialogue`)
- **What happened**: The extra argument is silently ignored. State has no name.
- **What I expected**: Either it should accept a name, or the docs should show the no-argument signature
- **Severity**: rough-edge
- **Notes**: The state API is `setAttribute`/`attributes`, `createDialogue`/`useDialogue`/`getDialogue`, and `createContextItem`/`getContext`. None of this is documented in the README. I had to read source code to figure out how to use it.

### 8. `createStateItem` requires a default value — error message is confusing

- **What I tried**: `createStateItem("myItem")` (no default value) then `item.setValue("hello")`
- **What happened**: Error: `"Invalid value type. Expected undefined, received string"` — because the default value was `undefined` and `setValue` checks `typeof value === typeof this.value`
- **What I expected**: Either accept any type when no default is given, or require the default value in the API and explain why
- **Severity**: confusing
- **File/line**: `src/state/item.ts:16-17` — the type check `typeof value === typeof this.value` means if you don't pass a default, you can never set a value
- **Notes**: The TypeScript signature `createStateItem<T>(name: string, defaultValue: T)` makes the second arg required, so this is really a JS-only issue. But the error message "Expected undefined" is baffling to a beginner.

### 9. Things that worked well

- `replaceStringTemplate` parser works correctly
- `createState()` + `setAttribute` + `createDialogue` + `createContextItem` all work correctly once you know the API
- `createStateItem` with a proper default value works great and enforces type consistency
- `state.serialize()` gives a clean JSON representation of all state
- `listToJson` works fine for simple key:value pairs without colons in values
