## Findings

### 1. Mock LLM provider key not documented anywhere obvious

- **What I tried**: Used `useLlm("openai.chat-mock")` as mentioned in CLAUDE.md ("Mock LLM providers: `openai.chat-mock` / `openai.mock`")
- **What happened**: `LlmExeError: Invalid provider: openai.chat-mock`
- **What I expected**: It should work since the docs say to use that key
- **Actual working key**: `openai.chat-mock.v1` (found by reading source code at `src/llm/config/openai/index.ts:48`)
- **Severity**: confusing
- **File/line**: CLAUDE.md line mentioning "Mock LLM providers" is inaccurate. Also `src/llm/config/openai/index.ts:48`
- **Notes**: The README doesn't mention mock providers at all. As a beginner I had no idea how to test without real API keys. I had to read the source code to find the right key. The error message says "Provide a valid provider" but doesn't list what valid providers are.

### 2. Boolean parser only accepts literal "true" - very surprising

- **What I tried**: `createParser("boolean").parse("yes")` and `createParser("boolean").parse("1")`
- **What happened**: Both return `false`. Only the exact string `"true"` (case-insensitive) returns `true`.
- **What I expected**: A boolean parser designed for LLM output should handle common affirmative responses like "yes", "Yes", "1", "yeah", etc. LLMs frequently respond with "yes" or "no" when asked yes/no questions.
- **Severity**: rough-edge
- **File/line**: `src/parser/parsers/BooleanParser.ts:17` - the parse method only checks `clean === "true"`
- **Notes**: This is technically "working as coded" but feels like a trap. If I tell an LLM "respond with yes or no" and pipe it through the boolean parser, everything comes back `false`. The docs don't explain that only "true"/"false" are accepted.

### 3. README lists parsers incompletely

- **What I tried**: Looked at README "Built-In Parsers" section to find all available parsers
- **What happened**: README lists: `stringExtract`, `listToJson`, `listToArray`, `markdownCodeBlock`, and then `...etc`
- **What I expected**: A complete list, or at least a link to one
- **Actual full list** (from `src/parser/_functions.ts:216-236`): `json`, `markdownCodeBlocks` (plural), `markdownCodeBlock`, `listToArray`, `listToJson`, `listToKeyValue`, `replaceStringTemplate`, `boolean`, `number`, `stringExtract`, `string`
- **Severity**: confusing
- **Notes**: The `...etc` is frustrating as a beginner. I didn't know `string`, `boolean`, `number`, `replaceStringTemplate`, `markdownCodeBlocks`, or `listToKeyValue` existed until I read source code. CLAUDE.md mentions more of them but uses slightly different names.

### 4. JSON parser silently returns `{}` on invalid input

- **What I tried**: `createParser("json").parse("This is not JSON at all")`
- **What happened**: Returns `{}` with no error
- **What I expected**: Either throw an error or return null/undefined so I know parsing failed
- **Severity**: rough-edge
- **Notes**: For a beginner debugging why their LLM output isn't being parsed correctly, getting `{}` silently is very confusing. You'd think your LLM returned an empty object, not that parsing failed.

### 5. README overview example uses undefined `mySchema`

- **What I tried**: Followed the README overview example that shows `createParser("json", { schema: mySchema })`
- **What happened**: `mySchema` is never defined in the example. I had no idea what shape a schema should be.
- **What I expected**: Either define `mySchema` in the example, or link to docs explaining what a schema is
- **Severity**: confusing
- **Notes**: I found `defineSchema` exists by looking at exports in `src/index.ts`, but the README never mentions it. I eventually figured out you can pass a plain object and it works, but I was guessing.

### 6. Everything that DID work well

- `createChatPrompt` with Handlebars templating worked perfectly. `{{#each}}` and `{{input}}` substitution was smooth.
- `createPrompt("chat", ...)` and `createChatPrompt(...)` both work fine.
- `createDialogue` worked exactly as the README shows.
- `createCustomParser` was easy and intuitive.
- `executor.on("onSuccess", ...)` hooks fired correctly with useful data (including start/end times, input, output, metadata).
- `stringExtract` parser was great - gave it a sentence with "yes" buried in it and correctly extracted "yes".
- `listToArray` correctly parsed markdown-style lists.
- `number` parser correctly extracted numbers from sentences like "The answer is 42."
- `markdownCodeBlock` parser correctly extracted code and language from fenced code blocks.
- The full pipeline (prompt -> mock LLM -> parser -> result) worked smoothly once I had the right mock key.
- The `listToKeyValue` parser works well for `key: value` formatted output.
