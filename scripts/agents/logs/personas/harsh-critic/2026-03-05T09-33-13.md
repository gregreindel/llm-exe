## Findings

### 1. README lists `createParser("listToJson")` — docs show wrong API for custom parser

**What I tried**: Read the README "Built-In Parsers" section and tried to understand what parser types are available.

**What happened**: README line 75 lists `createParser("listToJson")` which is correct. But `examples/claude.md` (line ~103) shows this custom parser API:
```ts
const customParser = createParser({
  name: "customParser",
  parse: (input: string) => { ... }
});
```

**What I expected**: This is completely wrong. `createParser` takes a string type as the first argument, not an object. The actual API for custom parsers is `createCustomParser("name", fn)`. Someone following `examples/claude.md` will get a type error immediately.

**Severity**: confusing
**File/line**: `examples/claude.md:103`

### 2. examples/claude.md shows Zod schema with `createParser("json")` — actual API uses JSON Schema

**What I tried**: Followed the "Parser Selection" example in `examples/claude.md`.

**What happened**: The example shows:
```ts
const jsonParser = createParser("json", {
  schema: z.object({ result: z.string(), confidence: z.number() }),
});
```

**What I expected**: The actual `schema` option expects a JSON Schema object (via `json-schema-to-ts`), not a Zod schema. The library exports `defineSchema()` for this. A developer following this doc will pass a Zod object and get bizarre runtime behavior or type errors.

**Severity**: confusing
**File/line**: `examples/claude.md:91`

---

### 3. Prompt uses `add*Message` but Dialogue uses `set*Message` — inconsistent verb

**What I tried**: Used both `ChatPrompt` and `Dialogue` to build message sequences.

**What happened**: ChatPrompt has `addUserMessage()`, `addAssistantMessage()`, `addSystemMessage()`. Dialogue has `setUserMessage()`, `setAssistantMessage()`, `setSystemMessage()`. Both are doing the same thing — appending a message to an array.

**What I expected**: Same verb for the same operation. "set" implies replacing; "add" implies appending. Both actually append. This will trip up anyone switching between the two. Pick one and stick with it.

**Severity**: rough-edge
**File/line**: `src/state/dialogue.ts:22` vs `src/prompt/chat.ts:125`

### 4. `setMessageTurn` adds system message AFTER assistant message

**What I tried**: Read the `Dialogue.setMessageTurn()` method.

**What happened**: It calls `setUserMessage`, then `setAssistantMessage`, then `setSystemMessage`. So a "turn" puts the system message last — after the assistant reply. That's bizarre message ordering for any LLM API.

**What I expected**: If you're going to have a convenience method for a "turn", system should come first (if included at all). Or honestly, don't include system in a "turn" method — system messages aren't part of conversational turns.

**Severity**: rough-edge
**File/line**: `src/state/dialogue.ts:135-143`

### What works well

- **The Prompt -> LLM -> Parser -> Executor pattern is clean.** I get it immediately. The README example communicates the value prop in about 30 seconds. Solid.
- **Mock LLM (`openai.chat-mock.v1`) works great.** It echoes input back, so you can test the full pipeline without API keys. The mock scenario tests are comprehensive and well-structured.
- **Type inference through createParser is genuinely good.** The overload structure in `_functions.ts` means you get the right parser type back. The effort put into this is evident.
- **Hooks API is well-designed.** `on`/`off`/`once`/`clearHooks` with hook limits. The metadata passed to hooks is rich. The `getHookCount` for monitoring is a nice touch.
- **`createDialogue().addFromOutput()` is smart.** Bridging the gap between raw LLM output and conversation history automatically is the kind of ergonomic win that matters.
- **Test coverage is solid.** 1275+ tests. Mock scenario file covers the full pipeline. This is a library that takes testing seriously.
