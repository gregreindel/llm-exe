- Added `validateInput` method that throws a clear `TypeError` (naming the actual runtime type and the executor's name/type) when called with primitives, arrays, or `null`. `undefined` is allowed and coerced to `{}` for convenience.
- Removed the silent `ensureInputIsObject` coercion from `getHandlerInput`. The utility itself is preserved for explicit use by `callable.ts`, which intentionally wraps LLM tool-call string args.

## Next Steps
- Pre-existing lint errors (14 total) live in `src/utils/modules/json.ts`, `src/utils/modules/pick.ts`, and Handlebars helper files — `no-undef`, `no-empty`, `no-prototype-builtins`. Worth a separate cleanup pass.
- `LlmExecutor.execute` still does `_input ?? {}` before delegating to `super.execute`, which silently coerces `null`/`undefined` to `{}`. This is now the only remaining implicit coercion and is arguably a UX nicety (prompts with no variables). If the maintainer wants strict-runtime parity for `LlmExecutor` too, that would be a follow-up — and probably a breaking change for callers who rely on the null/undefined affordance.
- `callable.ts` still wraps via `ensureInputIsObject` before calling `_handler.execute`. With validation now strict, that wrapping is the only path that preserves the historical "string-in" behavior for tool-call args. Worth documenting that this wrapping is intentional, since the rest of the executor surface no longer coerces.
